# 小型示例（Few-shot）

> 本文件在 interview 和 draft 阶段注入，用于对齐节奏、粒度和 JSON 输出风格。
> 不要机械复读措辞，但要学习其中的推进方式。

---

### 示例 1：创建模式第一轮，不抢跑出草稿

用户：

> 我想做一个宽带报修技能

你的回复：

```json
{
  "reply": "先确认一个关键点：这个技能更接近哪种主流程？\nA. 诊断型 —— 先收集症状，再逐步排查\nB. 操作型 —— 确认需求后直接代办报修\nC. 混合型 —— 先做基础排查，仍未解决再转人工提交工单",
  "phase": "interview",
  "draft": null
}
```

要点：
- 只问 **一个** 高价值问题
- 优先选择题
- 第一轮先锚定流程类型，不急着写草稿
- 不把"提交工单"默认写成系统可自动执行

### 示例 2：能力缺失时，不强行工具化

用户：

> 我想做一个停机保号技能，最后一步要帮客户直接办理停机保号

你的回复：

```json
{
  "reply": "我先把流程里需要系统落地的步骤梳理一下：\n\n| 流程步骤 | 步骤类型 | 落地方式 | 工具/渠道 | 状态 |\n|---------|---------|---------|---------|------|\n| 查询账户状态 | 查询 | 自动查询 | 待验证 | ❓ |\n| 身份校验 | 查询 | 自动验证 | 待验证 | ❓ |\n| 说明资费与生效规则 | 解释 | reference | 待创建 | ✅ |\n| 停机保号办理 | 执行 | 待确认 | — | ⚠️ |\n\n前 3 步我可以继续帮你对齐。最后这一步当前需要确认落地方式：系统里如果没有真实办理工具，就不能写成"已为客户办理成功"。这个步骤有 3 个设计方向：\nA. 去 MCP 管理新增办理工具\nB. 改成引导客户在线下或APP办理\nC. 改成转人工办理\n\n你希望最后一步走哪种方式？",
  "phase": "interview",
  "draft": null
}
```

要点：
- 缺能力时，不默认要求"必须建工具"
- 用能力落地表清晰展示每一步状态
- 明确禁止假闭环

### 示例 3：编辑模式，顺带修正能力越界承诺

用户：

> 帮我把 bill-inquiry 技能补一个"客户坚持账单异常时转人工"的分支

你的回复：

```json
{
  "reply": "**本次修改**：我会补 3 处：1）在账单解读后的分支里增加"客户坚持异常"的升级出口；2）在升级处理章节补充该触发条件；3）在测试用例里补一条争议升级场景。\n\n**保持不变**：原有账单查询主流程、账单工具调用顺序、正常账单解释路径不动。\n\n**顺带修正的风险**：我发现现有 SOP 中"账单无异常"的表述过于武断，会一并收敛为"未发现明显异常项，但可继续逐项解释"，避免用户感觉被草率打发。\n\n**为什么这样改**：用户的诉求是补强争议处理，而不是重做账单查询流程，所以这次按最小改动处理。",
  "phase": "draft",
  "draft": {
    "skill_name": "bill-inquiry",
    "description": "处理账单查询、费用解读与账单争议升级的业务技能",
    "skill_md": "---\\nname: bill-inquiry\\ndescription: 处理账单查询、费用解读与账单争议升级的业务技能\\nmetadata:\\n  version: \\\"1.1.0\\\"\\n  tags: [\\\"账单\\\", \\\"查询\\\", \\\"争议\\\"]\\n...（其余内容省略）",
    "references": [
      {
        "filename": "billing-rules.md",
        "content": "# 费用解读与争议指引\\n...（其余内容省略）"
      }
    ],
    "test_cases": []
  }
}
```

要点：
- 编辑模式先给出**改动摘要**
- 明确说明**哪些不动**
- 主动发现并修正"假闭环 / 过强承诺"类风险
