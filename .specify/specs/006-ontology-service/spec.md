## 系统概述

当前项目已经在往 `CDP`、`Interaction Platform`、`Work Order`、`Agent Workspace` 等多个基础模块演进，但系统仍存在一个更深层的问题：

- 数据散落在多个微服务与数据库中
- 不同系统对同一业务对象采用不同字段与不同口径
- 运营规则存在于人、文档、脚本和分散配置里，难以解释与复用
- 系统可以“看报表”，但很难在强约束下给出可执行、可审计、可回滚的动作方案

因此，本规格定义一个新的基础模块：

> **Ontology Service**
> = **企业运行时语义内核**
> = **统一业务对象模型** + **规则与约束层** + **事件与状态投影层** + **函数与推演层** + **受控执行编排层**

它不是又一个报表库，也不是一个只给 LLM 喂数据的中间层。  
它的职责是把多个微服务中的离散数据，提升为统一的业务语言、运行时语义与受控动作闭环，使系统从“看见问题”走向“理解问题、评估方案、执行动作”。

## 业务目标

本规格希望解决以下结构性问题：

- 让 `Event / Queue / Skill / Agent / Ticket / Customer` 等对象在全局拥有统一语义，而不是散落在不同系统字段中
- 让“对象、属性、关系、状态、事件、规则、动作”成为一套可计算、可治理、可版本化的模型体系
- 让客服中心、运营平台、AI Agent、Hybrid UI 都可以消费同一套语义层，而不是各自拼装上下文
- 让运营应急场景具备完整闭环能力：`听得懂 -> 算得清 -> 做得出`
- 让执行动作经过权限、规则、灰度、审计和回滚机制，而不是让 LLM 直接改生产配置

因此，`Ontology Service` 的正式定位应是：

> **连接“分散业务事实”与“受控运营动作”的运行时语义层**

## 用户场景与测试

### 用户故事 1 — 系统能统一识别事件影响面并定位牵涉资源（Priority: P1）

当客服中心发生网络故障、账单异常、系统降级等突发事件时，平台需要快速识别受影响的队列、渠道、技能、座席、客户与工单，而不是让运营经理手工翻系统对口径。

**为什么是 P1**：如果系统连“发生了什么、影响了谁、牵扯哪些资源”都说不清，就谈不上后续推演与执行。

**独立测试**：给定一个 `Event` 与来自多个源系统的同步数据，系统能返回受影响的 `Queue / Channel / Skill / Agent / Customer / Ticket` 关系图，并解释其关联依据。

**验收场景**：

1. **Given** 一个“账单扣费异常”事件被同步进入本体层，**When** 平台请求影响面分析，**Then** 系统应返回关联的账单类队列、相关技能、当前在线座席和高风险客户集合。
2. **Given** 一个事件同时影响语音与在线消息渠道，**When** 查询事件拓扑，**Then** 系统应能显示 `Event -> Channel -> Queue -> Skill -> Agent` 的关系链，而不是只返回一张孤立事件表记录。

---

### 用户故事 2 — 系统能在规则约束下生成可比较的运营方案（Priority: P1）

当排队激增、SLA 逼近阈值时，运营负责人需要看到多套可比较的方案，以及每套方案对 SLA、AHT、放弃率、投诉风险和成本的量化影响。

**为什么是 P1**：只有能在约束条件下做多方案评估，Ontology 才不是“知识图谱展示层”，而是真正的决策支撑层。

**独立测试**：输入实时到达率、AHT、在线座席、技能覆盖和事件优先级，系统可生成至少两套备选方案，并返回量化预测与命中规则说明。

**验收场景**：

1. **Given** 某个高优先级事件导致未来 30 分钟排队预计翻倍，**When** 平台请求运营方案评估，**Then** 系统应输出“先保 SLA”和“先控成本”等可比较方案，并附带预测指标。
2. **Given** VIP 保护、跨技能认证与渠道优先级规则已配置，**When** 方案生成器运行，**Then** 系统不应输出违反约束的支援或分流建议。

---

### 用户故事 3 — 系统能把方案转成受控动作草案并完成写回校验（Priority: P1）

当运营主管选择某个方案后，平台需要把该方案转成结构化动作草案，并在权限、规则、灰度和回滚校验通过后，回写到实际业务系统。

**为什么是 P1**：这一步决定平台是否只是“会分析”，还是能够真正落到执行。

**独立测试**：选择一个运营方案后，系统能生成 `ActionDraft`，并将校验通过的动作写回 `WFM / CTI / KM / CRM / Work Order` 等目标系统。

**验收场景**：

1. **Given** 方案要求提升队列优先级并抽调特定技能座席支援，**When** 进入执行阶段，**Then** 系统应先生成动作草案并执行规则、权限、灰度检查，而不是直接改动源系统。
2. **Given** 某个动作因权限不足或违反规则被拒绝，**When** 执行校验完成，**Then** 平台应返回失败原因、受影响动作和可审计记录，而不是部分黑盒写回。

---

### 用户故事 4 — 模型变更、规则变更与推演结果必须可版本化和可审计（Priority: P2）

当业务对象、规则、函数或动作模板发生变更时，平台需要保留版本、差异、上线时间与审计信息，确保运营决策的解释链条完整。

**为什么是 P2**：没有版本治理，本体层会很快退化成新的配置黑洞。

**独立测试**：修改规则或对象模型后，系统能够记录版本、变更人、发布时间，并在后续方案解释中引用生效版本。

**验收场景**：

1. **Given** VIP 优先级规则被调整，**When** 新规则上线后再次生成运营方案，**Then** 方案说明中应标明命中的规则版本。
2. **Given** 某次执行导致业务副作用，**When** 平台回溯决策链条，**Then** 能看到参与计算的对象快照、规则版本、函数结果和执行记录。

## 边界情况

- 同一业务主体在不同微服务中存在不同主键、不同名称或不同生命周期时，系统如何完成身份映射与实例合并？
- 某些源系统只有定时同步而无实时事件时，系统如何标记语义投影的时效性与置信度？
- 多条规则相互冲突，或规则与动作模板发生矛盾时，系统如何选择拒绝执行、降级执行或升级人工审批？
- 某个写回动作执行成功，而后续联动动作失败时，系统如何补偿、回滚或标记人工接管？
- 某次事件造成瞬时高频变更时，系统如何避免语义层与推演层被事件风暴拖垮？
- `m9-ui-model` 等界面模型如何与业务本体协同，而不污染核心对象语义？

## 需求

### 功能需求

- **FR-001**: 系统必须新增一个独立的 `ontology_service`，作为跨微服务的运行时语义内核，而不是把本体能力散落在各个业务服务中。
- **FR-002**: `ontology_service` 必须明确区分 `TBox` 与 `ABox`，前者负责对象类型、关系类型、规则类型和动作类型定义，后者负责实例、实例关系、运行时状态与事件事实。
- **FR-003**: `ontology_service` 必须支持统一的对象建模，至少覆盖对象、属性、关系、状态、事件、规则、函数和动作八类核心语义。
- **FR-004**: `ontology_service` 必须从多个微服务的表和事件中同步数据，并构建统一实例视图，而不要求源系统共享数据库。
- **FR-005**: `ontology_service` 必须支持 identity map、实例归并、关系投影和聚合根识别，以表达跨系统业务对象的统一身份。
- **FR-006**: `ontology_service` 必须把行为、规则和事件作为一等建模对象，而不应只停留在静态对象关系建模。
- **FR-007**: `ontology_service` 必须支持规则的版本化、审计和显式解释，且规则必须能被方案评估与执行校验共同消费。
- **FR-008**: `ontology_service` 必须支持函数与计算能力，用于排队预测、负载评估、成本质量权衡和多方案模拟。
- **FR-009**: `ontology_service` 必须能够输出结构化方案集，每个方案都包含动作建议、量化预测、命中规则与风险说明。
- **FR-010**: `ontology_service` 必须把执行输出定义为 `ActionDraft`，LLM 或 Planner 只能生成草案，不能直接绕过治理层写回生产系统。
- **FR-011**: `ontology_service` 必须在写回前执行权限、规则、灰度、审批与回滚校验，并保留完整审计记录。
- **FR-012**: `ontology_service` 必须支持事件驱动运行时，使对象状态、关系投影和派生事实可以随源事件持续更新。
- **FR-013**: `ontology_service` 必须向 `Interaction Platform`、`CDP`、`Work Order`、`Agent Workspace`、运营控制台与 AI Agent 提供统一语义服务接口。
- **FR-014**: `ontology_service` 必须明确源系统仍是事务真值来源，本体层不接管订单、账单、工单、排班或路由配置的原始所有权。
- **FR-015**: `ontology_service` 的 V1 存储设计必须优先支持关系型元数据、事件流和图投影混合架构，而不强制要求一开始采用纯 RDF / triple store。
- **FR-016**: `ontology_service` 必须将界面模型与业务本体分层治理，允许 `Hybrid UI / m9-ui-model` 复用本体语义，但不与核心运行时模型混存。
- **FR-017**: `ontology_service` 必须支持可回放、可重建和可重算，确保实例投影、推演结果和动作草案在必要时可以基于事件与规则版本重建。
- **FR-018**: `ontology_service` 的第一阶段必须覆盖“客服中心运营应急”场景，并支持从事件识别、影响分析、方案评估到动作草案输出的完整闭环。

## 成功标准

### 可量化指标

- **SC-001**: 对于客服中心运营应急场景，平台必须能在统一语义层中表达 `Event / Queue / Skill / Agent / Customer / Ticket` 的影响关系，而不是依赖人工跨系统汇总。
- **SC-002**: 在给定事件、实时队列状态和规则约束的情况下，平台必须能生成不少于两套可比较方案，并输出对应的量化预测指标与命中规则说明。
- **SC-003**: 任一执行动作都必须可追溯到对应的方案、规则版本、函数计算结果、审批记录与写回结果，不允许出现无法解释的生产动作。
- **SC-004**: 源系统新增一个对象来源表、一个事件源或一个新关系类型时，不应要求重写所有消费端逻辑，只应在本体模型和投影层完成扩展。
- **SC-005**: 本体层输出给 `AI Agent`、运营控制台与 Hybrid UI 的核心对象语义必须保持一致，避免每个消费方维护自己的“伪语义层”。
- **SC-006**: 在写回受控模式下，任何未通过权限、规则或灰度校验的动作都不得直接改动下游生产系统。
