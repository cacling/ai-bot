## 系统概述

基于 **Vercel AI SDK** 构建的智能电信客服 Agent 全栈应用，Agent 名为**小通**，提供**文字客服**、**语音客服**与**坐席工作台**三种交互模式。

核心设计理念是将"知"与"行"分离：
- **Skills（知识层）**：按需加载领域知识
- **MCP Tools（执行层）**：连接电信业务系统，执行实际操作

## 用户场景与测试

### 用户故事 1 — 账单查询（P1）

用户通过文字或语音向小通询问话费、账单明细。系统并行加载 bill-inquiry Skill 和 query_bill MCP 工具，返回结构化账单卡片。

**优先级理由**: 话费查询是电信客服最高频场景，用户量最大。

**独立测试**: 发送"查话费"，验证返回 bill_card 卡片和费用明细。

**验收场景**:

1. **前提** 用户已登录（手机号注入 system prompt），**当** 发送"查话费"，**则** 返回当月账单明细卡片（含 total、plan_fee、data_fee 等字段）
2. **前提** 用户查询指定月份，**当** 发送"查 2026 年 1 月账单"，**则** 返回该月账单
3. **前提** 查询月份无数据，**当** 查询不存在的月份，**则** 回复"未找到该月账单"

---

### 用户故事 2 — 业务退订（P1）

用户要求退订增值业务。系统加载退订政策 Skill，查询当前已订业务，向用户确认后执行退订。

**优先级理由**: 涉及不可逆操作，必须确保流程安全、用户确认。

**独立测试**: 发送"退订视频会员"，验证确认流程和 cancel_card 卡片。

**验收场景**:

1. **前提** 用户已订阅视频会员，**当** 发送"退订视频会员"，**则** Agent 先确认业务名称和费用影响，用户确认后执行退订
2. **前提** 用户未订阅目标业务，**当** 请求退订，**则** 返回"未订阅该业务"

---

### 用户故事 3 — 套餐咨询（P2）

用户咨询套餐内容或升级方案。系统并行加载套餐知识和当前用户信息，提供对比推荐。

**优先级理由**: 套餐咨询频次高，支持交叉销售。

**独立测试**: 发送"我想升级套餐"，验证返回 plan_card 对比卡片。

**验收场景**:

1. **前提** 用户当前使用 50G 套餐，**当** 询问升级，**则** 返回可用套餐列表及对比分析

---

### 用户故事 4 — 网络故障诊断（P2）

用户报告网络问题。系统加载诊断流程 Skill 和 diagnose_network 工具，执行逐步诊断并返回结果卡片。

**优先级理由**: 用户体验直接影响满意度，需快速定位问题。

**独立测试**: 发送"网速很慢"，验证返回 diagnostic_card 和诊断步骤。

**验收场景**:

1. **前提** 用户流量接近用尽，**当** 报告网速慢，**则** 诊断结果指出流量用量高并建议升级

---

### 用户故事 5 — 语音客服（P2）

用户通过语音与小通交互。系统代理 GLM-Realtime WebSocket，支持全程免唤醒实时语音对话、MCP 工具调用与智能转人工。

**优先级理由**: 语音是电信客服传统核心渠道。

**独立测试**: 建立语音连接，语音说"查话费"，验证语音回复和工具调用。

**验收场景**:

1. **前提** 语音连接已建立，**当** 用户说"查话费"，**则** GLM 调用 query_bill 工具并语音回复账单信息
2. **前提** 用户情绪激烈，**当** 情绪检测为"愤怒"，**则** 推送 emotion_update 事件

---

### 用户故事 6 — 坐席工作台（P3）

人工客服通过坐席工作台实时监控客户对话。系统通过 Session Bus 同步对话内容，异步触发情感分析和转人工摘要。

**优先级理由**: 辅助人工客服提效，非直接面客。

**独立测试**: 客户侧发送消息，验证坐席工作台实时收到消息和情感分析。

**验收场景**:

1. **前提** 坐席已连接工作台，**当** 客户发送消息，**则** 坐席实时收到消息 + emotion_update 事件
2. **前提** 客户触发转人工，**当** transfer_data 事件到达，**则** 坐席收到 handoff_card 摘要

---

### 用户故事 7 — 外呼语音（P3）

系统通过语音外呼进行催收或营销。机器人主动发起开场白，按话术流程与客户交互。

**优先级理由**: 外呼是增量业务场景，非核心服务。

**独立测试**: 建立外呼连接，验证机器人自动开场白和工具调用。

**验收场景**:

1. **前提** 催收任务 C001，**当** 建立外呼连接，**则** 机器人自动说催收开场白并可调用 record_call_result 工具

---

### 用户故事 8 — 知识管理（P3）

管理员通过知识库编辑器管理 Skills：创建、编辑、版本控制、沙箱测试、灰度发布。

**优先级理由**: 后台管理功能，支撑运营效率。

**独立测试**: 创建新版本，编辑 SKILL.md，沙箱测试，发布到生产。

**验收场景**:

1. **前提** 管理员在编辑器中，**当** 创建新版本并编辑，**则** 版本快照保存到 .versions/
2. **前提** 版本已保存，**当** 执行沙箱测试（mock 模式），**则** 返回测试结果
3. **前提** 测试通过，**当** 发布版本，**则** 文件复制到 biz-skills/ 并自动生效

### 边界情况

- 用户手机号不存在时，query_subscriber 返回 `{"found": false}`
- MCP Server 连接失败时，后端返回 HTTP 500
- Agent 执行超过 180 秒时，AbortSignal 中断
- 退订未订阅的业务时，cancel_service 返回失败信息
- GLM-Realtime 连接断开时，等待 pendingHandoff 最长 20 秒

## 需求

### 功能需求

- **FR-001**: 系统**必须**支持文字客服、语音客服和坐席工作台三种交互模式
- **FR-002**: 系统**必须**将领域知识（Skills）与执行能力（MCP Tools）分层设计
- **FR-003**: 系统**必须**按需懒加载 Skills，不在启动时全量注入
- **FR-004**: 系统**必须**在执行不可逆操作（如退订）前向用户确认
- **FR-005**: 系统**必须**支持在单个 ReAct 步骤中并行调用工具
- **FR-006**: 系统**必须**在 Agent 执行时实时高亮 Mermaid 流程图
- **FR-007**: 系统**必须**支持基于渠道的技能路由（online、voice、outbound-collection、outbound-marketing）
- **FR-008**: 系统**必须**支持技能版本管理（完整目录快照）
- **FR-009**: 系统**必须**检测高风险变更并触发审批流程
- **FR-010**: 系统**必须**支持合规关键词过滤（AC 自动机，< 1ms）
- **FR-011**: 系统**必须**支持多语言语音服务（zh/en），非中文走翻译回退
- **FR-012**: 系统**必须**提供回归测试能力，支持 6 种断言类型（contains、not_contains、tool_called、tool_not_called、skill_loaded、regex）

### 待澄清

- **FR-013**: 系统**必须**妥善处理 GLM-Realtime WebSocket 断连 [待澄清：重连策略未定义 — 指数退避自动重连？通知用户手动重连？最大重试次数？]
- **FR-014**: 系统**必须**在生产环境对所有 API 和 WebSocket 请求进行认证 [待澄清：认证方案未确定 — X-User-Id header（当前方案）、JWT、OAuth2 还是 session-based？WebSocket 升级请求如何认证？]
- **FR-015**: 系统**必须**支持生产级数据库部署 [待澄清：SQLite → PostgreSQL 迁移路径未规划 — 时间线、迁移工具、兼容过渡期、触发迁移的数据量阈值？]
- **FR-016**: 外呼语音**必须**符合电信行业法规 [待澄清：工信部通话时间限制是否适用？是否需要限制外呼时段（如禁止 21:00-8:00）？单客户最大外呼频次？是否需要在通话开头披露 AI 身份？]
- **FR-017**: 系统**必须**定义支持的浏览器兼容范围 [待澄清：MediaSource API（语音播放）和 BroadcastChannel（跨标签页同步）需要现代浏览器 — 最低版本？移动端浏览器支持？Safari 对 MediaSource 的兼容性？]
- **FR-018**: 系统**必须**明确并发会话行为 [待澄清：同一手机号是否可以同时有多个活跃会话？若可以，Session Bus 事件如何路由？若不可以，新会话启动时现有会话如何处理？]
- **FR-019**: 系统**必须**定义 Agent 达到 maxSteps 上限时的行为 [待澄清：Agent 在 10 步内未解决查询时用户看到什么？自动转人工？通用"无法处理"提示？换策略重试？]
- **FR-020**: 系统**必须**具备提示词注入防护 [待澄清：如何处理对抗性用户输入？传入 LLM 前是否有输入净化？除 system-prompt 约束外是否有其他防护措施？]

### 核心实体

- **Subscriber**: 电信用户（手机号、套餐、余额、流量用量）
- **Skill**: 领域知识模块（SKILL.md + references/ + scripts/）
- **MCP Tool**: 外部系统操作工具（query_subscriber、cancel_service 等 15 个）
- **Session**: 对话会话（session_id + 消息历史）
- **SkillVersion**: 技能版本快照（status: saved/published）

## 成功标准

### 可量化指标

- **SC-001**: 账单/套餐查询端到端响应时间 ≤ 5 秒，退订/诊断 ≤ 8 秒（基于 Step-3.5-Flash，单用户无并发负载）
- **SC-002**: 所有测试用例通过，文件覆盖率 ≥ 90%
- **SC-003**: 合规关键词匹配延迟 < 1ms
- **SC-004**: Skills 修改后零停机生效
- **SC-005**: 退订操作 100% 经过用户确认
