# 业务 Skill 编写规范 — 检查清单

> 本文件在 confirm 阶段注入，用于审查草稿是否符合规范。
> 标注 `[AUTO]` 的项已由脚本自动校验，你不需要重复检查——只需关注未标注的项。

---

## 新建 Skill 检查清单

### 目录与文件

- [AUTO] 目录名符合 kebab-case 命名
- [AUTO] 参考文档与状态图 `%% ref:` 交叉校验一致
- [ ] `references/` 至少包含一个有意义的参考文档（不是空文件）

### SKILL.md Frontmatter

- [AUTO] `name`、`description`、`metadata.version`、`tags`、`mode`、`trigger`、`channels` 全部由脚本校验
- [ ] `description` 语义准确，能帮助意图路由正确匹配（脚本只检查非空，不检查语义）

### SKILL.md 章节

- [AUTO] 必要章节存在性和顺序
- [AUTO] 状态图 `%% tool:` 节点后有 `<<choice>>` 分支
- [AUTO] 状态图有起始/终止状态
- [AUTO] 全局转人工出口检测
- [AUTO] outbound 合规门控和呼叫结果分支检测
- [ ] 角色定义是否准确反映业务场景（脚本只检查存在性）
- [ ] `## 触发条件` 的关键词是否覆盖了用户的典型表述
- [ ] `## 工具与分类` 中的分类映射表是否完整
- [ ] 操作指引后有"问题是否解决"确认反馈环（脚本暂不检查）
- [ ] 合规关键路径是否建模完整（DND、情绪升级、操作前确认）
- [ ] `## 升级处理` 表格是否覆盖真实场景（不超过 10-15 行）
- [ ] `## 合规规则` 是否包含该业务的专属合规项
- [ ] `## 回复规范` 是否包含禁止跳步约束和节奏规则

### 回复模板（assets/）

- [AUTO] 操作类工具多次调用时检测是否需要 assets
- [AUTO] 是否包含"每次只调用一次"的显式约束
- [ ] assets 模板的话术骨架是否合理、字段引用是否正确

### 合规

- [ ] 包含数据来源合规项（禁止凭空捏造数据）
- [ ] 包含操作确认合规项（变更操作前须用户确认）
- [ ] 包含隐私保护合规项（不索要完整身份证号等）
- [ ] outbound 模式包含通话录音告知和拨打时段限制

### 运行时可编译性（推荐）

- [ ] 关键节点有 `%% step:<id>` 稳定标识
- [ ] 工具节点有 `%% kind:tool`，确认节点有 `%% kind:confirm`
- [ ] choice 节点的出边有 `%% guard:` 注释
- [ ] 确认节点有 `user.confirm` + `user.cancel` 两条出边
- [ ] 工具节点后有 `tool.success` + `tool.error` 分支
