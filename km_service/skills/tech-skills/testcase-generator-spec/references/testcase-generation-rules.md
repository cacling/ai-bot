# 测试用例生成详细规则

> 本文档在 Stage 2（Testcase Synthesis）时加载，补充 SKILL.md 中的生成规则。

---

## 1. turns 编写规则

### 单轮 vs 多轮

| 场景类型 | turns 数量 | 说明 |
|---------|-----------|------|
| 简单查询（查账单、查套餐） | 1 | 单条用户输入即可覆盖 |
| 确认型流程（退订、办理） | 2-3 | 需要触发 → 确认 → 完成 |
| 多步诊断（故障排查） | 2-4 | 描述问题 → 追问 → 方案 → 升级 |
| 意图切换 | 2 | 先说 A 再改成 B |

### 用户消息风格

- 用自然语言，不要用测试术语
- 模拟真实用户的表达习惯（口语化、有错别字、省略主语）
- 同一技能的不同 case 应使用不同的表达方式

好的示例：
- "帮我查一下这个月话费"
- "我想退掉那个视频包"
- "网不好 信号差"

差的示例：
- "请执行账单查询功能"
- "测试退订流程"
- "触发故障诊断技能"

## 2. assertions 编写规则

### 断言精准度

| 级别 | 示例 | 问题 |
|------|------|------|
| 过于宽泛 | `contains: "的"` | 几乎任何回复都通过 |
| 适当 | `contains: "账单"` | 验证回复与查账相关 |
| 过于严格 | `contains: "您本月账单金额为128.50元"` | 依赖具体数值，换用户就挂 |

### 推荐的断言组合

**功能测试**：`tool_called` + `contains`（验证工具被调用且回复有相关内容）
**边界测试**：`contains` 或 `response_mentions_any`（验证系统能合理响应）
**异常测试**：`tool_not_called` + `contains`（验证不会误操作，且给出合理引导）
**状态测试**：多轮中用 `contains` 分别验证确认前后的回复差异

### 不要滥用 regex

正则断言应保留给需要模式匹配的场景（如验证回复包含金额格式 `\d+\.?\d*元`）。
简单的关键词匹配用 `contains` 即可。

## 3. priority 分配规则

| Priority | 含义 | 分配给 |
|----------|------|--------|
| 1 | 核心路径，必须通过 | 主功能正常流程、核心工具链路 |
| 2 | 重要分支，应该通过 | 确认拒绝、常见边界、转人工 |
| 3 | 补充场景，最好通过 | 罕见边界、特殊错误、风格验证 |

P1 用例应覆盖 skill 最核心的 1-2 条路径。一个 skill 通常有 2-4 个 P1 用例。

## 4. persona_id 使用规则

仅当测试需要特定用户状态时才使用：

| persona_id | 适用场景 |
|------------|---------|
| `normal_user` | 正常用户（有套餐、有余额） |
| `arrears_user` | 欠费用户 |
| `new_user` | 新用户（无历史数据） |
| `suspended_user` | 已停机用户 |

大多数 case 不需要指定 persona_id，引擎会使用默认用户。

## 5. coverage_matrix 规则

- 每条 REQ-xxx 必须至少被一个 TC-xxx 覆盖
- 若某条需求确实无法设计合理的自动化测试，在 coverage_matrix 中标记但不强制生成 case
- coverage_matrix 用于事后审计，不影响执行
